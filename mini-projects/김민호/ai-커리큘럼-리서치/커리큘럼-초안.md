# AI 시대의 코드 판단력 — 커리큘럼 초안 아웃라인

## 연구 질문

> AI가 코드를 대신 써주는 시대에, 그 코드의 옳고 그름을 판단하는 선구안은 어떻게 기르는가?

---

## 1. 학습자 프로필

### 김민호

| 항목 | 현재 상태 |
|---|---|
| 코드 읽기 | Python 단어 수준 유추. 프레임워크 코드 판단 어려움 |
| 원리 지식 | if/else, function, OOP 개념. 자료구조/알고리즘 기초 부족 |
| AI 도구 | Claude Code 2개월+, Gemini CLI, 3대 LLM Pro 구독 |
| 실전 경험 | Doran SaaS (Next.js/PostgreSQL/Gemini/Docker), 30세션, E2E 62/62 pass |
| 투자 시간 | 풀타임, 주 20시간 |
| 핵심 갭 | 실전은 앞서가는데 원리가 따라오지 못함 |

### 오뱀(오범석)

| 항목 | 현재 상태 |
|---|---|
| 코드 읽기 | 대충 흐름은 보임 |
| 원리 지식 | 거의 모름 (개념만) |
| AI 도구 | 범용(ChatGPT 등) 다 써봄. 개별 도구 경험 없음 |
| 실전 경험 | Excel VBA 자동화 → Claude Code 프로그램 제작 → 사내 ERP 구축 |
| 투자 시간 | 주 6~8시간 |
| 핵심 갭 | 만들 수 있지만 왜 되는지/안 되는지 판단이 어려움 |
| 목표 | 대체불가능한 아이덴티티 |

### 공통점과 차이점

**공통**: 둘 다 AI로 실무 결과물을 만들어본 경험이 있다. 둘 다 원리 지식이 부족하다. 둘 다 "돌아가니까 OK"에서 "구조적으로 맞는지 판단"으로 올라가고 싶다.

**차이**: 투자 시간(20시간 vs 6~8시간), 도메인(SaaS vs 인증 자동화), AI 도구 숙련도(높음 vs 낮음).

## 2. 설계 원칙

리서치에서 추출한 5가지 원칙:

### 원칙 1: 원리 → 도구 순서 (Level 0이 전제조건)

CS146S가 자료구조/OS를 전제조건으로 요구하고, 케냐 실험이 "기초 없는 AI 사용은 역효과"를 실증했다. 원리 없이 도구를 쓰면 "마법사의 제자 함정"에 빠진다.

**적용**: 원리 기초를 먼저 채운 후 AI 판단력 훈련으로 넘어간다.

### 원칙 2: 비교 훈련이 판단력의 핵심 방법론

CS146S W7이 보여준 패턴 — 수동 리뷰 vs AI 리뷰 비교 → 차이 분석 → 개인 휴리스틱 개발. 판단력은 추상적 이론이 아니라 반복적 비교 행위를 통해 성장한다.

**적용**: 매 모듈에 "내 판단 vs AI 판단" 비교 활동을 포함한다.

### 원칙 3: 문서화가 메타인지를 기른다

CS146S의 writeup.md — "AI가 한 것과 내가 한 것의 경계를 의식적으로 구분하는 훈련". 이 문서화 행위 자체가 판단력 훈련이다.

**적용**: 모든 학습 활동에 짧은 회고 기록을 남긴다.

### 원칙 4: "먼저 만들기"로 동기 부여 + "왜 되는지" 이해로 심화

Andrew Ng의 "Go and build stuff"와 "마법사의 제자 함정" 사이의 균형. 빠른 성과 경험이 동기를 부여하지만, 이해 없이 쌓으면 위험하다.

**적용**: 이미 만든 것(Doran, ERP)을 돌아보며 원리를 역추적하는 방식. 백지에서 시작하지 않는다.

### 원칙 5: 2인 페어는 자산이다

CS146S의 코드 리뷰, CMSC 398Z의 페어 코딩 + 토론 — 혼자보다 둘이 판단력이 더 빨리 성장한다. 서로의 코드를 리뷰하는 것 자체가 훈련.

**적용**: 주간 페어 리뷰 세션을 커리큘럼의 핵심 활동으로 설계한다.

## 3. 역량 위계와 현재 위치

리서치에서 발견한 역량 위계:

```
Level 0: 원리 기초 — 자료구조, 알고리즘, 프로그래밍 기본 개념
Level 1: AI 도구 사용 — 프롬프팅, IDE 통합 도구 활용
Level 2: AI 도구 내부 이해 — MCP, 프로토콜, 에이전트 작동 원리
Level 3: AI 워크플로 설계 — 자동화, 오케스트레이션, 가드레일
Level 4: AI 출력 판단 — 코드 리뷰, 보안 감사, 기술 부채 평가
Level 5: 독립적 제품 구축 — "만들지 말아야 할 것" 판단 포함
```

### 현재 위치 진단

| 레벨 | 민호 | 오뱀 |
|---|---|---|
| Level 0 | 부분적 (개념 수준, 실습 부족) | 부분적 (개념만) |
| Level 1 | 능숙 (Claude Code 일상 사용) | 초급 (범용 AI만) |
| Level 2 | 부분적 (MCP 설정 경험) | 없음 |
| Level 3 | 부분적 (하네스/스킬 설계) | 없음 |
| Level 4 | 없음 (핵심 갭) | 없음 (핵심 갭) |
| Level 5 | 부분적 (Doran, 하지만 판단력 없이) | 부분적 (ERP, 하지만 판단력 없이) |

**진단**: 둘 다 Level 1/3/5를 실전으로 건너뛴 상태. Level 0(원리)과 Level 4(판단)가 빈 채로 결과물을 만들고 있다. 이것이 정확히 DORA 2025가 보여준 패턴 — "PR은 늘지만 성과는 안 느는" 상태.

## 4. 커리큘럼 구조

### 전체 로드맵

```
Module 0: 역추적 (4주)      ← "이미 만든 것"에서 원리를 발견한다

Module 4: 통합 프로젝트 (4주) ← 자기 프로젝트에 적용
```

**총 26주 (약 6개월)**

| | 민호 (주 20시간) | 오뱀 (주 6~8시간) |
|---|---|---|
| 개인 학습 | 15시간 | 4~6시간 |

## 5. 모듈 상세

### Module 0: 역추적 (4주)

> "이미 만든 것에서 원리를 발견한다"

**목적**: 백지에서 시작하지 않는다. Doran과 ERP에서 이미 사용한 개념을 의식적으로 인식한다.

**활동**:
- 자기 프로젝트의 코드를 AI 없이 읽어보고, 이해되는 것과 안 되는 것을 구분
- AI에게 "이 코드가 뭘 하는지 설명해줘"라고 묻고, 설명이 맞는지 스스로 판단해보기
- 판단이 안 되는 영역을 목록으로 정리 → Module 1의 개인화된 학습 목표가 됨

**페어 세션**: 서로의 프로젝트 코드를 교환하여 읽어보기. "어디서 막히는지"가 원리 갭을 드러낸다.

**산출물**: 각자의 "모르는 것 목록" + "이미 아는 것 목록"

**참고 자료**: 없음 (자기 프로젝트가 교재)

---

### Module 1: 원리 기초 (8주)

> "빠진 Level 0을 채운다"

**목적**: 코드를 판단하려면 최소한의 원리 지식이 필요하다. 전부를 배우는 것이 아니라, 판단력의 토대가 되는 핵심만 채운다.

**핵심 주제 (주차별)**:

| 주차 | 주제 | 왜 필요한가 | 참고 자료 |
|---|---|---|---|
| W1-2 | 함수, 분해(decomposition), 추상화 | 코드 구조를 판단하는 기본 단위 | CS106A 강의 + 과제 |
| W3-4 | 자료구조 (리스트, 딕셔너리, 스택, 큐) | AI가 선택한 자료구조가 적절한지 판단 | CS50 W5 + CS106A |
| W5-6 | 알고리즘 기초 (탐색, 정렬, Big-O) | "이 코드가 느린 이유"를 판단하는 눈 | CS50 W3 |
| W7 | 데이터베이스/SQL 기초 | Doran(PostgreSQL), ERP 모두 DB 사용 | CS50 W7 |
| W8 | 네트워크/API 기초 | 웹앱과 서비스 간 통신 판단 | CS50 W8-W9 |

**활동**:
- CS50/CS106A 강의 시청 → 핵심 개념 정리
- 매주: 자기 프로젝트에서 해당 개념이 사용된 곳 찾기 ("역추적 연결")
- 매주: 간단한 연습 문제 (CS50 문제 세트 중 선택)

**페어 세션**: 주간 개념 퀴즈 + "내 프로젝트에서 이 개념은 여기에 쓰였다" 공유

**참고 자료**:
- CS50 (YouTube 무료, 완전 독학 가능) — 원리 중심, 초보 대상
- CS106A (공개 자료 풍부) — Python 기반, 분해/추상화 강조
- 두 과정을 통째로 따라가는 것이 아니라, 필요한 주제만 발췌

---

### Module 2: 코드 읽기 (4주)

> "남의 코드를 읽고 판단하는 눈"

**목적**: 코드를 "쓰는" 것이 아니라 "읽는" 역량을 집중 훈련한다. Level 4(판단)의 직전 단계.

**핵심 주제**:

| 주차 | 주제 | 활동 |
|---|---|---|
| W1 | 코드 읽기 기초 | 오픈소스 프로젝트의 README → 엔트리포인트 → 핵심 로직 추적 |
| W2 | 패턴 인식 | 반복되는 코드 구조 (CRUD, 미들웨어, 에러 핸들링 등) 식별 |
| W3 | 코드 리뷰 입문 | GitHub PR을 읽고, 변경의 의도와 영향을 파악 |
| W4 | 자기 프로젝트 리뷰 | Doran/ERP 코드를 "남의 코드"처럼 읽고 리뷰 |

**페어 세션**: 서로의 프로젝트 코드를 리뷰. "이 부분이 왜 이렇게 되어 있는지" 질문하기.

**참고 자료**:
- UCSD 과정 7 "Working with Large Code Bases" 개념 참고
- CS146S W7 코드 리뷰 과제 구조 참고

---

### Module 3: AI 판단력 (6주) — 핵심 모듈

> "AI가 만든 코드를 비판적으로 평가하는 힘"

**목적**: 이 커리큘럼의 존재 이유. CS146S W6-W7의 방법론을 자기 프로젝트에 적용한다.

**핵심 주제**:

| 주차 | 주제 | 활동 | 참고 자료 |
|---|---|---|---|
| W1 | AI 출력 비교 훈련 | 같은 문제를 AI에게 풀게 하고, 자기도 풀어보고, 비교 | CS146S W7 방법론 |
| W2 | 오탐 식별 | AI 코드에서 "돌아가지만 틀린" 부분 찾기 | CS146S W6 (Semgrep) |
| W3 | 기술 부채 판단 | AI 코드의 "좋은 부채 vs 나쁜 부채" 구분 | Andrew Ng 프레임 |
| W4 | 보안 감각 | AI 코드의 보안 취약점 탐지 (인젝션, 인증 등) | CS146S W6 |
| W5 | 자율성 수준 설정 | AI에게 어디까지 맡길지 의식적으로 결정하고 기록 | CS146S writeup.md |
| W6 | 종합 리뷰 실습 | 자기 프로젝트(Doran/ERP)에서 AI가 만든 코드를 전면 리뷰 | 전체 통합 |

**핵심 활동 — "비교 노트"**:
- 매주 1개 이상의 코딩 문제를 선정
- Step 1: 자기가 먼저 해결 시도 (코드 or 의사코드)
- Step 2: AI에게 동일 문제를 맡김
- Step 3: 두 결과를 비교 — AI가 더 나은 점, 내가 더 나은 점, AI가 놓친 점
- Step 4: 비교 노트 작성 (writeup.md 형식)

**페어 세션**: 서로의 비교 노트를 공유. "나는 이걸 놓쳤는데 네가 잡았다" — 관점의 교차가 판단력을 키운다.

---

### Module 4: 통합 프로젝트 (4주)

> "자기 프로젝트에 적용한다"

**목적**: Module 0~3에서 기른 눈을 실제 프로젝트에 통합한다.

**민호**: Doran 코드 전면 리뷰 + 리팩토링 계획 수립
**오뱀**: ERP/인증 자동화 코드 전면 리뷰 + 개선 계획 수립

**활동**:
- W1-2: 자기 프로젝트 코드를 Module 3의 프레임으로 전면 리뷰
- W3: 리뷰 결과를 바탕으로 개선 계획 작성 (리팩토링/보안/성능)
- W4: 페어 발표 — 서로에게 "내 프로젝트의 문제점과 개선 계획" 발표

**산출물**: 각자의 프로젝트 리뷰 보고서 + 개선 계획서

## 7. 페어 세션 설계

주 1회 2시간, 고정 시간.

### 세션 구조 (2시간)

```
[10분] 체크인 — 이번 주 뭘 배웠는지 한 문장으로
[40분] 상호 리뷰 — 이번 주 과제/코드/노트를 교환하여 리뷰
[30분] 토론 — 의견 차이가 나는 지점에 대해 논쟁
[20분] 다음 주 계획 — 각자 뭘 할지 합의
[10분] 회고 기록 — 오늘 새로 알게 된 것 1가지 적기
[10분] 여유
```

### 핵심 규칙

- **속도 맞추기 강제 안 함**: 민호가 Module 2를 하고 있을 때 오뱀이 아직 Module 1이어도 된다. 페어 세션에서는 각자의 현재 위치에서 서로에게 배운다.
- **가르치는 것이 최고의 학습**: 민호가 먼저 배운 것을 오뱀에게 설명하면, 민호도 더 깊이 이해한다.
- **비교 노트 교환 필수**: Module 3에서는 서로의 "AI vs 나" 비교 노트를 교환하고 피드백한다.

| 5 | 서로의 코드를 리뷰하고 개선점을 제안할 수 있다 | 상대방 코드에 대해 구체적 피드백 3개 이상 |
